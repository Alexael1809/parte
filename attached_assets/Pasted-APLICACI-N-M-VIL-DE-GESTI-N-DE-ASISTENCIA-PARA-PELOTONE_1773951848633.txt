APLICACIÓN MÓVIL DE GESTIÓN DE ASISTENCIA PARA PELOTONES (REPLIT + SUPABASE)

📱 Descripción General

Aplicación móvil nativa para Android (desarrollada en Flutter o React Native para facilitar su creación en Replit) que permite la gestión de asistencia de aproximadamente 790 personas organizadas en pelotones de tamaño variable (9 a 50 personas). La aplicación debe funcionar sin conexión a internet (offline-first) y sincronizar automáticamente con Supabase al restablecer la conexión.

El desarrollo se realizará íntegramente en Replit, con el código fuente alojado en un repositorio Git, y utilizando Supabase como backend (autenticación, base de datos PostgreSQL, almacenamiento y funciones en tiempo real).

🛠️ Tecnologías Propuestas

· Frontend Móvil: Flutter (recomendado para compilar a APK nativo con buena experiencia offline) o React Native (Expo).
· Backend y Base de Datos: Supabase (PostgreSQL, Autenticación, Row Level Security, Realtime).
· Almacenamiento Local: SQLite (con sqflite en Flutter o react-native-sqlite-storage) para modo offline.
· Sincronización: Mecanismo propio con lógica de conflictos (última modificación gana).
· Autenticación: Supabase Auth con JWT, más biometría (huella digital) en el dispositivo.
· Notificaciones Push: Firebase Cloud Messaging (FCM) integrado con Supabase.

👥 Usuarios y Roles

Usuario Estándar (Recolector)

· Usa la app en su celular.
· Registra asistencia del pelotón asignado por el superusuario.
· Puede trabajar sin conexión a internet.
· Ve únicamente su pelotón asignado.
· Puede acceder al plan de búsqueda de cualquier persona de su pelotón (ver, editar y compartir).

Superusuario (Administrador)

· Control total del sistema.
· Gestiona usuarios, procesos, pelotones y personas.
· Visualiza y exporta reportes consolidados.
· Ve el total de TODOS los pelotones (estadísticas globales).
· Único con permisos para editar asistencias pasadas.
· Puede acceder al plan de búsqueda de cualquier persona de cualquier pelotón.

🏗️ Estructura Organizacional

Procesos

· Máximo 3 procesos activos simultáneamente.
· Formato ejemplo: "Proceso 1-2025", "Proceso 2-2025", "Proceso 1-2026".
· Posibilidad de archivar procesos anteriores.

PNFs (Programas Nacionales de Formación)

· Fijos iniciales: Criminalística, Investigación Penal, Servicio Policial, Servicio Ciencias de Fuego.
· Posibilidad de editar nombres a futuro (solo superusuario).

Pelotones

· Tamaño variable (9 a 50 personas).
· Asignados a un Proceso y un PNF específico.
· Ejemplo: Pelotón "A" del Proceso "1-2025" del PNF "Investigación Penal".

Personas

· Datos básicos: nombres, apellidos, cédula de identidad, sexo, pelotón asignado.
· Plan de búsqueda: 3 números de teléfono, dirección y lugar de origen (lugar de residencia).
· Historial de asistencia vinculado a la persona (aunque cambie de pelotón).

4. MÓDULO DE PLAN DE BÚSQUEDA (Nuevo)

Acceso

· En la lista de asistencia (tanto para usuario estándar como superusuario), al presionar sobre una persona se abre una ventana modal o pantalla con el título "Plan de Búsqueda".

Contenido de la Ventana

· Nombres y apellidos (grande, destacado).
· Cédula de Identidad.
· Estado actual (Presente, Inasistente, Comisión, Reposo) con su color correspondiente.
· Teléfonos (3 campos): mostrados como "Teléfono 1: +58 XXX XXX XXXX", etc.
· Dirección: calle, ciudad, etc.
· Lugar de origen (lugar de residencia, puede ser igual a la dirección o diferente).
· Botones:
  · Editar: permite modificar teléfonos, dirección y lugar de origen.
  · Compartir: abre las opciones nativas de compartir (WhatsApp, email, etc.) con un texto formateado que incluye toda la información del plan de búsqueda.
  · Cerrar.

Permisos

· Usuario estándar: puede ver, editar y compartir el plan de búsqueda de las personas de su pelotón.
· Superusuario: puede ver, editar y compartir el plan de búsqueda de cualquier persona (desde la lista de personas o desde el drill down de inasistentes).

Edición

· Al tocar "Editar", se habilitan campos para modificar los 3 teléfonos, la dirección y el lugar de origen.
· Validación básica (números de teléfono, campos no vacíos).
· Guardado local y sincronización con Supabase.

Compartición

· El texto compartido debe tener un formato legible, por ejemplo:
  ```
  PLAN DE BÚSQUEDA
  Nombre: María González
  CI: 12.345.678
  Estado: Inasistente
  Teléfonos: 0412-1234567, 0416-7654321, 0424-9876543
  Dirección: Av. Principal, Casa N° 10, Caracas
  Lugar de origen: Barquisimeto, Estado Lara
  ```
· Se puede compartir como texto plano o generar una imagen/PDF (opcional).

🗄️ Base de Datos y Sincronización (Supabase)

Base de Datos Local (Offline)

· Motor: SQLite (sqflite en Flutter).
· Tablas locales: réplica de las tablas necesarias de Supabase (usuarios, personas, asistencias, pelotones, procesos, pnfs, planes_busqueda).
· Sincronización bidireccional:
  · Al abrir la app con internet: descarga últimos datos (personas, pelotones, etc.).
  · Al tomar asistencia offline: guarda local con flag sync_status = 'pending'.
  · Al recuperar internet: sincroniza en segundo plano (sube asistencias pendientes y baja actualizaciones).
  · Resolución de conflictos: última modificación gana (basada en timestamp).

Base de Datos Remota (Supabase)

· Tablas principales:
  · usuarios: id, email, rol, peloton_id (si es estándar), activo, etc.
  · procesos: id, nombre, activo, fecha_archivado (nullable).
  · pnfs: id, nombre.
  · pelotones: id, nombre, proceso_id, pnf_id.
  · personas: id, nombres, apellidos, ci, sexo, peloton_id, created_at, updated_at.
  · planes_busqueda: id, persona_id, telefono1, telefono2, telefono3, direccion, lugar_origen.
  · asistencias: id, persona_id, fecha, estado, motivo (nullable), usuario_id, created_at, updated_at.
  · historial_cambios: id, tabla, registro_id, usuario_id, cambio, timestamp.
· Autenticación: Supabase Auth (email/contraseña).
· Seguridad: Row Level Security (RLS) para que cada usuario vea solo sus datos (su pelotón) y el superusuario todo.
· Realtime: Opcional para actualizaciones en vivo del dashboard.
